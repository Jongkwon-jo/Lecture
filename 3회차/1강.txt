슬라이드 1: 표지 (1분)
안녕하세요, 여러분. 국립창원대학교 PRU 재직자 교육과정에 참여해주셔서 감사합니다. 오늘은 'AI-Agent란 무엇인가'라는 주제로 여러분과 함께 시간을 보내게 되었어요.

이 강의는 단순한 대화형 AI에서 실제로 업무를 수행하는 AI로의 진화 과정을 다룹니다. 우리가 지금까지 사용해왔던 챗봇과는 전혀 다른 차원의 기술이죠. 실무자와 관리자 여러분 모두에게 유익한 내용이 될 거예요.

슬라이드 2: 교육 목차 (1분 30초)
오늘 우리가 함께 탐험할 여정을 미리 살펴볼까요? 총 6개의 세션으로 구성되어 있습니다.

먼저 진화의 시작점부터 살펴보겠습니다. 대화형 AI가 어떻게 업무 수행 AI로 변모했는지 그 패러다임의 변화를 이해하는 것이죠. 두 번째로는 AI-Agent의 핵심 정의와 네 가지 구성요소를 파헤쳐볼 거예요.

세 번째 세션에서는 ReAct, Planner-Executor 같은 작동 원리를 다룹니다. 네 번째는 실제로 Agent가 해결할 수 있는 문제 유형들이고요. 다섯 번째는 도입 로드맵과 산업별 사례를 살펴보며, 마지막으로 핵심 내용을 정리하고 질의응답 시간을 갖겠습니다.

슬라이드 3: 학습 목표 (1분 30초)
이 과정을 마치고 나면 여러분이 달성하게 될 핵심 역량들입니다. 주목해주세요.

첫째, 단순 챗봇과 업무 수행형 Agent를 명확히 구분할 수 있게 됩니다. 둘째, Agent의 4대 구성요소인 계획, 메모리, 도구, 행동을 정확히 설명할 수 있죠.

셋째는 여러 아키텍처 패턴의 특징을 비교 분석하는 능력입니다. 넷째, 실무 조직에서 Agent 기술을 적용하기 적합한 문제를 식별할 수 있어요. 마지막으로 성공적인 도입을 위한 로드맵과 평가 지표를 제시할 수 있게 됩니다.

슬라이드 4: Section 01 전환 (30초)
자, 그럼 본격적으로 시작해볼까요? 첫 번째 섹션입니다.

대화형 AI에서 업무 수행 AI로의 진화 과정을 살펴보겠습니다. '질문-답변'이라는 단순한 패러다임에서 '목표-계획-실행'이라는 새로운 시대로 넘어가는 거죠.

슬라이드 5: 대화형 LLM vs 업무 수행 Agent 비교 (2분)
이 슬라이드가 오늘의 핵심입니다. 왼쪽과 오른쪽을 비교해보세요.

대화형 LLM은 단발성 상호작용에 머물러 있어요. 입력에 대해 즉각적인 텍스트 응답만 생성하고, 대화가 끝나면 컨텍스트가 초기화되죠. 반면 업무 수행 Agent는 목표 지향적으로 계획을 수립합니다. 추상적인 목표를 설정하면 스스로 하위 작업을 나눠서 단계별로 실행하는 거예요.

도구 접근에서도 차이가 극명합니다. 대화형 LLM은 외부 세계와 단절되어 브라우징이나 API 호출 같은 실제 행동을 하지 못해요. 하지만 Agent는 브라우저, 데이터베이스, 사내 API를 직접 호출해서 실제 데이터를 조작하고 결과를 얻어냅니다.

상태 기억 능력도 다르죠. 대화형 모델은 긴 작업 흐름을 기억하는 능력이 부족합니다. 반면 Agent는 작업 수행 결과를 메모리에 저장하고, 오류 발생 시 스스로 수정하며 반복적으로 개선해나가요. 마지막으로 주체성의 차이입니다. 대화형 모델은 수동적이지만, Agent는 정의된 정책 범위 내에서 자율적으로 판단하고 행동합니다.

슬라이드 6: Agent의 정의 (2분)
그렇다면 Agent를 정확히 어떻게 정의할 수 있을까요?

환경을 인지하고, 목표 달성을 위해 계획하고 행동하며, 결과를 반성하여 스스로 개선하는 자율 시스템입니다. Perceive, Plan, Act, Reflect라는 네 단계의 순환 구조로 작동하죠.

네 가지 핵심 특징을 살펴보겠습니다. 첫째, 목표 지향성이에요. 구체적인 지시 없이도 "매출 분석해줘"라는 추상적인 목표를 이해하고 하위 작업을 생성합니다.

둘째는 도구 활용 능력입니다. 언어 모델의 한계를 넘어 브라우저 검색, API 호출, 코드 실행 등을 적극적으로 사용하죠. 셋째, 상태와 메모리 관리예요. 단발성 대화가 아니라 작업의 맥락을 기억하며 긴 호흡의 업무를 완결시킵니다.

넷째는 반복적 개선 루프입니다. 한 번의 실행으로 끝나지 않고, 결과를 스스로 평가하고 오류 시 계획을 수정해서 재시도하는 거죠.

슬라이드 7: Agent 핵심 구성요소 (2분)
Agent가 자율적으로 업무를 수행하려면 다섯 가지 핵심 요소가 필요합니다.

첫 번째, 목표와 지시사항이에요. 모든 행동의 기준점이 되는 부분이죠. 사용자가 전달한 목표나 시스템에 사전 정의된 목적을 말합니다.

두 번째는 지각과 컨텍스트입니다. 프롬프트 입력, RAG 검색 결과, 시스템 로그, 심지어 센서를 통해 환경을 인식하는 거예요. 세 번째는 도구죠. 브라우저, 데이터베이스, 사내 API, 코드 실행기 등 실제 작업을 수행하는 수단입니다.

네 번째, 메모리와 상태 관리예요. 대화 컨텍스트라는 단기 메모리와 지식 베이스라는 장기 메모리를 통해 일관성을 유지합니다. 마지막 다섯 번째는 정책과 계획이에요. 목표 달성을 위한 전략 수립과 단계별 계획 생성, 실행 제어 로직을 담당합니다.

슬라이드 8: 대표 아키텍처 패턴 (2분 30초)
이제 Agent가 문제를 해결하는 다양한 방식들을 살펴보겠습니다. 주요 패턴이 다섯 가지 있어요.

첫째, ReAct 패턴입니다. Reason과 Act의 합성어죠. 생각하고, 행동하고, 관찰하는 루프를 반복하면서 논리적으로 문제를 해결합니다. 가장 표준적인 방식이에요.

둘째는 Toolformer나 Function Calling이에요. 모델이 API나 도구 호출이 필요한 시점을 스스로 판단하고 실행하죠. OpenAI의 Function Calling이 대표적인 예입니다.

셋째, Planner-Executor 구조입니다. 계획 수립자와 실행자를 분리하는 거예요. 플래너가 전체 로드맵을 짜면 실행자가 각 단계를 수행해서 복잡도를 낮춥니다.

넷째는 Reflexion이나 Tree-of-Thoughts 같은 고급 기법이에요. 자기 반성을 통해 실패 원인을 분석하고 수정하거나, 여러 생각의 가지를 탐색해서 최적해를 찾습니다. 다섯째는 Multi-Agent 협업이죠. 개발자, 테스터, 기획자처럼 역할이 다른 여러 에이전트가 서로 대화하며 복합적인 업무를 완수합니다.

슬라이드 9: 대화형 AI의 한계점 (1분 30초)
그렇다면 왜 Agent가 필요한 걸까요? 기존 대화형 AI의 한계를 보면 이해가 됩니다.

첫째, 체계적 계획이 부재해요. 복잡한 다단계 업무를 스스로 세분화하고 순서대로 처리하는 능력이 없어서 단발성 답변에 그치죠.

둘째는 데이터 접근성 제한입니다. 실시간 웹 정보나 사내 보안 데이터베이스에 직접 접근할 수 없어서 최신성과 구체성이 떨어져요. 셋째, 장기 맥락 유지에 실패합니다. 대화가 길어지면 이전 지시사항을 잊거나 품질 일관성을 유지하지 못하는 거죠.

넷째는 실행 책임과 권한이 없다는 점입니다. 시스템을 제어하거나 변경 사항을 저장하는 실행 권한이 없어서 실제 업무를 완결 짓지 못해요. 단순 채팅 모델은 "실행"하지 못한다는 게 핵심입니다.

슬라이드 10: Agent의 등장 배경 (1분 30초)
그래서 Agent가 등장하게 된 배경을 살펴볼 필요가 있어요.

먼저 시장의 요구가 폭발적으로 증가했습니다. 단순 대화형 모델을 넘어 실질적인 업무 자동화와 신뢰할 수 있는 시스템에 대한 니즈가 커진 거죠.

기술적 성숙도도 높아졌어요. LLM의 복잡한 추론 능력이 고도화되고, LangChain 같은 프레임워크와 도구 생태계가 확장되어 기술적 구현이 용이해졌습니다. 비즈니스 자동화 수요도 중요한 요인이에요. 기존 RPA는 규칙 기반이라 예외 상황 처리와 비정형 데이터 분석에 한계가 있었거든요.

마지막으로 품질과 거버넌스 요구사항입니다. 기업 환경에 도입하려면 결과의 품질 관리, 행동 추적성, 보안과 안전 기준이 필수적이거든요. 이 세 가지 요인이 맞물리면서 Agent 시대가 열린 겁니다.

슬라이드 11: AI Agent 진화 타임라인 (1분 30초)
시간의 흐름 속에서 Agent의 진화 과정을 보겠습니다.

2022년, 대화형 LLM 시대죠. ChatGPT 등장으로 거대언어모델이 대중화되었지만 주로 단발성 질의응답에 초점이 맞춰져 있었어요.

2023년은 도구 사용과 추론의 시대입니다. RAG, ReAct 패턴, Function Calling을 통해 외부 데이터를 검색하고 도구를 사용하기 시작했죠. LangChain이 생태계를 주도했습니다.

2024년에는 멀티에이전트 협업이 성숙해졌어요. 단일 에이전트의 한계를 넘어 여러 역할이 협업하는 시스템이 나타났고, Autogen 같은 프레임워크가 등장했습니다. 그리고 2025년 이후는 오케스트레이션과 거버넌스 시대예요. 엔터프라이즈급 관리, 보안, 자율 운영 표준화가 진행되고 Agent 간의 거대 네트워크, 즉 IoA가 형성되고 있습니다.

슬라이드 12: Section 02 전환 (30초)
자, 이제 두 번째 섹션으로 넘어가겠습니다.

Agent는 어떻게 일하는가? 핵심 메커니즘인 Perceive, Plan, Act, Reflect를 중심으로 기본 워크플로우와 도구 사용 방법을 살펴보겠습니다.

슬라이드 13: 기본 워크플로우 (2분)
AI Agent가 작업을 수행하는 5단계 순환 과정을 이해하는 게 중요합니다.

1단계, 입력과 목표 해석이에요. 사용자의 모호한 요청을 시스템이 이해할 수 있는 구체적인 목표로 정의하고 변환하는 거죠.

2단계는 정보 수집입니다. RAG나 외부 검색을 통해 판단에 필요한 배경 지식과 데이터를 확보해요. 3단계, 계획 수립이죠. 최종 목표를 달성하기 위해 필요한 세부 작업들을 정의하고 우선순위를 배정합니다.

4단계는 실행입니다. API 호출, 데이터베이스 쿼리, 코드 실행 등 실제 도구를 사용해서 작업을 수행하는 거예요. 마지막 5단계는 검증과 반성입니다. 실행 결과를 평가하고 오류가 있다면 수정 계획을 수립하거나 최종 결과를 보고하죠.

이 다섯 단계가 순환하면서 Agent는 복잡한 업무를 완수합니다.

슬라이드 14: 도구 사용과 Function Calling (2분)
이제 코드로 실제 구현을 살펴볼게요. 이 파이썬 예제가 전체 프로세스를 보여줍니다.

2번 라인에서 목표를 설정합니다. "신제품 경쟁 분석 보고서 작성"이라는 추상적인 목표죠. 5번 라인에서 Agent가 계획을 수립해요. 목표를 달성하기 위해 필요한 하위 작업들을 순차적으로 나열하고 계획을 생성하는 겁니다.

7번부터는 각 단계를 반복하며 실행하는 루프예요. 9번 라인에서 환경과 단계를 인지하고, 12번에서 도구 사용이 필요한지 판단합니다. Function Calling을 통해 검색, 계산기, DB 조회 같은 외부 도구를 실행하고 결과를 얻죠.

16번 라인에서는 도구 실행 결과를 바탕으로 현재 상태를 추론하고, 필요시 계획을 수정하거나 다음 행동을 결정합니다. 마지막 20번 라인에서 모든 단계가 완료되면 수집된 정보와 추론 내용을 종합해서 사용자가 요청한 최종 산출물을 작성하는 거예요.

슬라이드 15: 메모리와 상태 관리 (2분)
Agent가 단순한 대화 모델과 다른 결정적인 이유가 바로 메모리와 상태 관리 능력입니다.

단기 메모리부터 보겠습니다. 현재 대화의 맥락, 즉 Context Window와 사용자의 즉각적인 지시사항을 저장해요. 작업 수행 과정에서 생성되는 중간 산출물, 즉 Thought를 일시적으로 유지하는 거죠.

장기 메모리는 좀 더 영구적입니다. 사내 위키, 매뉴얼, 과거 성공 사례 등을 벡터 데이터베이스에 저장해요. 필요할 때 검색해서 참조할 수 있는 영구적 지식 저장소인 거죠.

상태 모델링도 중요합니다. 복잡한 작업의 진행 단계를 To-do, In Progress, Done으로 관리해요. 오류 발생 시 처음부터 다시 시작하지 않고 체크포인트에서 재시도할 수 있게 하는 겁니다.

보안과 거버넌스도 필수예요. 메모리에 저장되는 데이터 중 개인정보나 민감 정보가 포함되지 않도록 마스킹 처리하거나 저장 기한을 설정해서 자동 파기하는 정책이 필요합니다.

슬라이드 16: 계획 기법 (1분 30초)
복잡한 목표를 달성 가능한 하위 작업으로 분해하는 계획 기법들을 살펴보겠습니다.

첫째, 체인 오브 소트예요. "단계별로 생각해보자"라는 프롬프팅을 통해 중간 추론 과정을 생성하죠. 복잡한 문제의 해결 능력을 극대화하는 기법입니다.

둘째는 트리나 그래프 탐색이에요. 여러 가지 해결 경로를 동시에 생성하고, 각 경로를 평가해서 유망하지 않은 경로는 가지치기하며 최적의 해답을 찾습니다.

셋째, 외부 플래너 연동입니다. 정해진 절차가 중요한 업무의 경우 BPMN 엔진이나 표준 운영 절차를 기반으로 하는 외부 시스템의 워크플로우를 따르도록 계획하는 거예요. 넷째는 제약 기반 계획이죠. 시간, 예산, API 호출 횟수 같은 자원 제약을 고려해서 가장 효율적인 실행 경로를 선택합니다.

슬라이드 17: 실행과 오류 복구 (1분 30초)
실행 단계에서 안전성을 보장하는 전략들입니다.

첫째, 폴백과 재시도예요. 도구 호출 실패 시 지수 백오프로 재시도하거나, 검색 엔진을 대체하는 등 미리 정의된 대안 경로로 우회하는 겁니다.

둘째는 엄격한 스키마 검증이죠. LLM이 생성한 함수 호출 인자가 올바른 JSON 형식인지 검증하고, 오류 발생 시 에러 메시지를 모델에 피드백해서 스스로 수정하게 합니다.

셋째, 부분 성공과 롤백입니다. 다단계 작업 중 중간 실패 시 트랜잭션처럼 이전 상태로 되돌리거나 실패 지점부터 안전하게 재개할 수 있도록 상태를 관리하는 거예요. 넷째는 인간 개입이죠. 비용 지출이나 데이터 영구 삭제 같은 고위험 작업 실행 전에는 반드시 인간의 승인을 받도록 게이트를 설정합니다.

슬라이드 18: 피드백 루프와 자기반성 (1분 30초)
Agent가 단순 실행을 넘어 품질을 개선하는 과정이 Reflection입니다.

첫째, 자기 평가 프롬프트예요. 결과물 생성 후 즉시 종료하지 않고 "이 답변이 정확한가?", "누락된 정보는 없는가?"라고 스스로에게 되묻는 검토 단계를 추가해서 품질을 높이는 거죠.

둘째는 다중 시도 합의입니다. 하나의 작업에 대해 여러 가지 버전의 해결책을 생성한 후 투표나 평가 모델을 통해 가장 우수한 결과를 최종 채택해요.

셋째, 실패 사례 학습이죠. 작업 실패나 오류가 발생했을 때 그 원인과 해결 과정을 장기 메모리에 기록하고, 향후 유사한 문제 발생 시 이를 참조해서 실수를 방지합니다. 넷째는 E2E 로그 분석이에요. 단순 결과뿐 아니라 에이전트의 생각과 행동 전체 과정을 로그로 남겨서 품질 저하의 원인을 분석하는 겁니다.

슬라이드 19: 안전·거버넌스 필수 점검사항 (2분)
AI Agent 도입 시 기능 구현만큼 중요한 게 안전과 통제입니다. 다섯 가지 필수 체크리스트를 보겠습니다.

첫째, 데이터 보안과 PII 마스킹이에요. 개인정보나 민감한 기업 데이터가 LLM으로 전송되기 전에 식별 불가능하게 마스킹 처리되었는지 확인해야 합니다.

둘째는 도구 권한 최소화 원칙이죠. Agent가 사용하는 API나 데이터베이스 접근 권한이 해당 업무 수행에 필요한 최소한의 범위로 제한되었는지 점검해야 해요.

셋째, 추적성과 감사 로그입니다. Agent의 사고 과정, 도구 실행 파라미터, 결과값이 모두 로그로 기록되어 사후 분석이 가능해야 하죠. 넷째는 사용 정책과 제한이에요. 금지된 행동 리스트, API 호출 횟수 제한, 예산 한도가 설정되어 있어야 합니다.

다섯째, 인간 승인 단계입니다. 데이터 삭제나 결제 실행 같은 고위험 작업 수행 전에 관리자의 승인을 받는 절차가 있어야 해요. 신뢰할 수 없는 Agent는 업무에 투입할 수 없다는 원칙을 기억하세요.

슬라이드 20: 평가와 모니터링 (2분)
Agent 시스템의 성공을 측정하는 세 가지 핵심 차원을 살펴보겠습니다.

정량적 지표부터 보죠. 정확도는 정답 일치율을 측정하고, 성공률은 Task 완수율을 봅니다. 비용은 토큰당 또는 호출당 비용이고, 지연시간은 End-to-End 응답 속도예요.

정성적 지표도 중요합니다. 사용자 만족도는 CSAT이나 NPS로 측정하고, 가독성은 답변의 명확성을 평가하죠. 일관성은 톤과 매너를 보고, 신뢰성은 환각이 없는지 확인하는 겁니다.

운영과 안전 지표도 필수예요. 실패율은 에러 퍼센티지를 추적하고, 재시도율은 루프 카운트를 봅니다. 도구 오류는 API 실패 건수를, SLA 준수율은 목표 달성 비율을 측정하죠.

평가 방법론은 세 가지입니다. Offline Benchmark는 정답셋을 이용한 자동 평가, Online A/B Test는 실제 트래픽 분산 비교, Human-in-the-Loop는 전문가의 직접 검토예요.

슬라이드 21: Section 03 전환 (30초)
이제 세 번째 섹션입니다. Agent가 해결하는 문제 유형을 살펴보겠습니다.

업무 가치와 자동화 난이도로 분류해서 적합한 적용 분야를 식별하는 게 목표예요. 정보 탐색부터 의사결정 지원까지 다양한 사례를 분석하겠습니다.

슬라이드 22: 케이스 스터디 01 - 정보 탐색·브라우징 (1분 30초)
첫 번째 문제 유형은 정보 탐색입니다. 흩어져 있는 웹이나 사내 문서에서 정보를 수집하고 종합해서 인사이트가 담긴 요약 보고서를 작성하는 업무죠.

입력으로는 자연어 질문, 경쟁사 목록, 검색 키워드, 타겟 URL 리스트가 들어옵니다. Agent의 행동은 다섯 단계예요. 검색 계획 수립, 다중 소스 검색 및 크롤링, 핵심 정보 추출, 내용 종합 및 요약, 출처 표기 및 보고서 생성이죠.

평가 지표는 세 가지입니다. 최신성은 얼마나 최근 정보를 포함했는지, 정확성은 팩트 체크가 되었는지, 출처 신뢰도는 공신력 있는 소스인지를 봐요. 리스크도 있습니다. 환각으로 없는 정보를 생성하거나, 저작권 이슈로 무단 스크래핑 문제가 발생하거나, 출처가 소실될 수 있죠.

슬라이드 23: 케이스 스터디 02 - 데이터 처리·ETL 보조 (1분 30초)
두 번째 유형은 데이터 처리입니다. CSV나 엑셀 같은 원본 데이터를 정제하고 변환해서 분석 가능한 형태로 가공하는 업무예요.

입력은 파일 경로, 데이터 스키마, 변환 규칙, 품질 기준입니다. Agent의 행동은 다섯 단계죠. 파일 파싱 및 구조 이해, 데이터 유효성 검증, 결측치와 이상치 정제, 파생 변수나 특징 생성, 결과 리포트 및 적재예요.

평가 지표는 오류율, 즉 변환 실패 건수와 처리 시간, 그리고 재현성입니다. 동일 입력 시 결과를 보장하는지가 중요하죠. 리스크는 스키마 추론 오류로 날짜나 숫자 형식을 잘못 인식하거나, 엑셀 서식이나 수식이 깨지거나, 잘못된 필터링으로 데이터가 유실되는 경우예요.

슬라이드 24: 케이스 스터디 03 - 워크플로 자동화 (1분 30초)
세 번째는 워크플로 자동화입니다. RPA를 확장한 개념이죠. 이메일, 티켓 시스템, ERP 등 여러 시스템을 오가는 복합 업무를 End-to-End로 자동 처리하는 유형이에요.

입력은 업무 규칙, 시스템 접근 권한, 트리거가 됩니다. Agent의 행동은 다섯 단계예요. 요청 수신 및 의도 분류, 담당 부서나 담당자 라우팅, 승인 요청 생성 및 발송, 데이터베이스나 ERP에 처리 결과 기록, 완료 알림 발송이죠.

평가 지표는 처리량, 즉 시간당 처리 건수와 SLA 준수, 그리고 자동화율입니다. 인적 개입 감소 비율이 중요해요. 리스크는 권한 남용으로 불필요한 시스템 접근이 발생하거나, 정의되지 않은 케이스에서 실패하거나, 에이전트 간 반복 호출로 무한 루프가 생기는 경우입니다.

슬라이드 25: 케이스 스터디 04 - 의사결정 지원·데이터 분석 (1분 30초)
네 번째는 의사결정 지원입니다. 다양한 원천의 데이터를 합성해서 여러 옵션을 비교 분석하고 최적의 권고안을 작성하는 업무죠.

입력은 핵심 KPI 지표, 예산이나 시간 제약, 후보 안, 과거 이력 데이터입니다. Agent의 행동은 다섯 단계예요. 데이터 수집 및 전처리, 시나리오 모델링, 옵션별 감도 분석, 최적안 도출 및 근거 마련, 최종 권고 보고서 생성이죠.

평가 지표는 설명가능성, 즉 판단 근거의 명확성과 투명성, 그리고 논리적 일관성입니다. 데이터 출처와 로직 공개가 중요하고, 모순 없는 추론 과정이 필요해요. 리스크는 편향으로 데이터 편향이 증폭되거나, 과도한 확신으로 불확실성을 무시하거나, 블랙박스로 추론 과정이 불투명해지는 경우입니다.

슬라이드 26: 케이스 스터디 05 - 고객지원·티켓 처리 (1분 30초)
다섯 번째는 고객지원입니다. 고객 문의를 자동으로 분류하고 답변을 생성하거나, 해결이 어려운 경우 정식 티켓을 생성해서 상담원의 처리를 보조하는 업무예요.

입력은 대화 로그, 지식베이스, SLA 정책, 고객 프로필입니다. Agent의 행동은 다섯 단계죠. 문의 유형 및 감정 분석, 지식베이스 조회(RAG), 답변 초안 생성, 자동 해결 또는 티켓 생성 판단, 상위 부서 이관이에요.

평가 지표는 1차 해결률, 즉 FCR과 고객 만족도인 CSAT, 그리고 처리 시간인 AHT입니다. 문의부터 해결까지 소요 시간이 중요하죠. 리스크는 민감 발화, 즉 욕설이나 긴급 상황에 오대응하거나, 중요 이슈를 방치하는 이관 누락, 개인정보 마스킹 실패예요.

슬라이드 27: 소프트웨어 에이전트 - Coding Agent (1분 30초)
특별한 유형으로 소프트웨어 에이전트가 있습니다. 복잡한 개발 작업을 계획, 테스트, 리팩토링 단계로 나누어 자동 수행하는 워크플로우예요.

코드 예제를 보겠습니다. 1번 라인에서 "레거시 코드 리팩토링"이라는 태스크를 설정하죠. 4번 라인부터 상세 작업 단계 계획 수립이 시작됩니다. 기존 로직 분석 및 테스트 케이스 생성, Linter로 코드 스타일 문제 자동 검출, 함수 분리 및 변수명 개선으로 리팩토링, 테스트 실행으로 기능 무결성 검증, 그리고 Pull Request 자동 생성이에요.

각 단계마다 도구를 사용합니다. Sandbox 환경에서 코드를 실행하고, 에러가 발생하면 Reflection을 통해 수정하고 재시도하죠. 최종적으로 자동 PR까지 완료합니다. 개발자의 생산성을 획기적으로 높일 수 있는 유형이에요.

슬라이드 28: Multi-Agent 협업 (1분)
여러 Agent가 협업하는 시스템도 중요합니다.

역할 분담이 핵심이에요. 연구자(Researcher)는 정보를 수집하고, 실행자(Executor)는 작업을 수행하며, 관리자(Manager)는 전체를 조율하죠.

협업의 이점은 세 가지입니다. 첫째, 병렬화예요. 여러 작업을 동시에 처리해서 속도를 높입니다. 둘째, 체크 앤 밸런스죠. 한 Agent의 결과를 다른 Agent가 검증해서 오류를 줄여요. 셋째는 환각 감소입니다. 여러 Agent가 교차 검증하면서 잘못된 정보 생성을 방지하는 거죠.

슬라이드 29: 산업 사례 - 제조/스마트 팩토리 (1분 30초)
실제 산업 적용 사례를 보겠습니다. 첫 번째는 제조업과 스마트 팩토리예요.

문제 상황은 이렇습니다. 다양한 센서 데이터와 매뉴얼이 분산되어 있어 이상 징후를 조기에 감지하기 어렵고, 작업 표준 절차가 문서화되지 않아 숙련 인력에 의존하는 거죠.

Agent 솔루션은 멀티모달 센서 데이터와 매뉴얼을 통합해서 이상 징후를 실시간 감지하고 알림을 발송합니다. 또한 작업 영상과 베테랑 작업자의 노하우를 분석해서 자동으로 SOP를 생성하죠.

효과는 명확합니다. 다운타임이 감소하고 리드타임이 단축돼요. 예방 정비가 가능해지고 품질 일관성이 향상되는 겁니다. 제조 현장의 효율성을 크게 높일 수 있어요.

슬라이드 30: 산업 사례 - 교육/창원대학교 (1분)
두 번째 사례는 교육 분야입니다. 바로 우리 창원대학교의 사례죠.

문제 상황은 학사 규정과 FAQ가 복잡하고 자주 변경되어 학생들의 반복적인 문의가 많다는 거예요. 행정 직원의 업무 부담이 크고, 야간이나 주말에는 응대가 불가능하죠.

Agent 솔루션은 학사 규정, 공지사항, FAQ를 벡터 DB에 저장하고 RAG 기반으로 정확한 답변을 제공합니다. 티켓 자동 생성으로 복잡한 문의는 담당자에게 연결하죠.

효과는 24시간 365일 가용성이 확보되고, 일관성 있는 답변 품질이 유지되며, 직원의 단순 반복 업무가 감소한다는 거예요. 학생 만족도도 향상됩니다.

슬라이드 31: AI Agent 도입 로드맵 (2분)
그렇다면 실제로 어떻게 도입해야 할까요? 3단계 로드맵을 제시합니다.

Step 01은 PoC 단계예요. 2주에서 4주 정도 소요되죠. 목표는 기술적 실현 가능성 검증입니다. 작은 범위의 파일럿 프로젝트로 시작하고, 단일 유스케이스를 선정해서 빠르게 프로토타입을 만들어요. 핵심 지표를 정의하고 기준선을 측정하는 게 중요합니다.

Step 02는 Pilot 단계입니다. 4주에서 8주 정도 걸려요. 목표는 실사용자 피드백 수집과 품질 개선이죠. 실제 업무 환경에 제한적으로 배포하고, 사용자 인터뷰와 로그 분석을 통해 개선점을 도출합니다. 안전장치와 모니터링을 강화하는 단계예요.

Step 03은 Scale 단계입니다. 지속적으로 진행되죠. 목표는 조직 전체로 확산과 거버넌스 체계 구축이에요. 여러 부서로 확장하고, 표준 운영 절차를 수립하며, 지속적인 모니터링과 개선을 진행합니다. 성공 사례를 공유해서 조직 문화를 변화시키는 거죠.

슬라이드 32: 기술 스택 선정 기준 (1분 30초)
기술 스택을 선택할 때는 두 가지 옵션이 있습니다.

첫째, Managed Service예요. OpenAI Assistants, Azure AI Studio, AWS Bedrock 같은 서비스죠. 장점은 빠른 시작과 인프라 관리 부담이 없다는 거예요. 단점은 비용이 높고 커스터마이징에 제약이 있으며 벤더 종속성이 생긴다는 점입니다.

둘째는 Self-Hosted나 Hybrid 방식이에요. LangChain, LlamaIndex, Autogen 같은 오픈소스 프레임워크를 사용하죠. 장점은 완전한 제어권과 비용 효율성, 그리고 온프레미스 배포 가능성이에요. 단점은 MLOps 인프라 구축 부담과 높은 기술 역량이 필요하다는 점입니다.

선정 기준은 네 가지예요. 데이터 민감도, 즉 클라우드 전송 가능 여부, 예산 규모와 트래픽, 확장성 요구사항, 그리고 조직의 MLOps 역량입니다. 이 네 가지를 종합적으로 고려해서 선택해야 해요.

슬라이드 33: 핵심 요약 (1분 30초)
오늘 배운 내용을 다섯 가지로 정리하겠습니다.

첫째, Agent의 핵심은 세 가지 요소입니다. 목표와 도구, 그리고 메모리죠. 이 세 가지가 유기적으로 결합되어 자율적인 업무 수행을 가능하게 해요.

둘째, 복잡도에 따라 패턴을 선택합니다. 단순한 작업은 ReAct로 충분하지만, 복잡한 작업은 Planner-Executor 구조가 필요하죠.

셋째, 문제 식별이 중요합니다. 모든 문제를 Agent로 해결할 수 없어요. 정보 탐색, 데이터 처리, 워크플로 자동화, 의사결정 지원 같은 적합한 유형을 선택해야 합니다.

넷째, 안전과 거버넌스는 필수예요. PII 마스킹, 권한 최소화, 감사 로그, 가드레일, 인간 승인 단계가 모두 갖춰져야 합니다. 다섯째, 평가와 운영이 핵심입니다. 정량·정성·운영 지표를 균형있게 모니터링하고 지속적으로 개선해야 해요.
