이겸
안녕하십니까, 여러분. 오늘 국립창원대학교 PRU 재직자 교육에 참석해 주셔서 감사합니다.
제가 오늘 여러분과 함께 나눌 주제는 'AI 자동화를 활용한 보고/문서 자동화 에이전트 실무'입니다.
품질, 생산, 정비 데이터의 흐름을 설계하고 반복적인 문서 작업을 지능형 템플릿으로 혁신하는 실전 가이드를 준비했습니다. 

오늘 교육을 통해 여러분이 얻어가실 핵심 내용은 크게 네 가지로 정리됩니다. 첫째, 제조 현장 보고/문서 자동화에 최적화된 에이아이 에이전트의 핵심 개념과 아키텍처 구조를 이해하게 되실 겁니다.
둘째, 품질·생산·정비 데이터가 발생해서 최종 보고서로 완성되기까지의 전체 자동화 프로세스를 단계별로 파악하실 수 있습니다. 셋째, 지능형 템플릿을 활용해 주간이나 월간 반복되는 보고 업무 시간을 획기적으로 줄이는 시스템 설계 방법을 익히게 되죠.
마지막으로, 실제 CSV 데이터를 활용해 차트를 그리고 PPT 및 워드 문서를 자동으로 생성하는 파이썬 코드를 직접 실습해볼 예정입니다.

전체 구성을 먼저 살펴보겠습니다. 오늘 교육은 크게 세 개의 섹션으로 나뉩니다.
첫 번째 섹션에서는 품질, 생산, 정비 보고서의 자동화 흐름을 다룹니다. 에이전트 개념, 아키텍처, 그리고 제조업 보고서 자동화 사례와 워크플로우 설계, 파이썬 실습을 진행하죠.
두 번째 섹션은 템플릿 기반 반복업무 절감 사례를 집중적으로 살펴봅니다. 템플릿 설계 방법론, RPA와 에이아이 에이전트 비교, 그리고 실제 적용 사례와 파이썬 실습이 포함돼 있습니다.
마지막으로 요약 및 핵심 정리로 강의를 마무리하겠습니다.

자, 첫 번째 섹션을 시작하겠습니다. 품질, 생산, 정비 보고서 자동화 흐름이라는 주제인데요.
이 섹션의 목표는 명확합니다. 현장 데이터를 기반으로 반복 보고서를 신뢰성 있게 자동화하는 방법을 설계하는 것이죠. 제조 현장의 실시간 데이터를 어떻게 수집하고 가공해서 의미 있는 보고서로 만들어낼 수 있는지 함께 알아보겠습니다.

보고/문서 자동화 에이전트란 정확히 무엇일까요? 단순한 매크로나 봇과는 차원이 다른 개념입니다.
첫째, 목표 기반 실행 방식을 갖추고 있습니다. 단순 반복 수행을 넘어서 보고목적과 KPI를 이해하고, 이를 달성하기 위한 단계별 실행 계획을 스스로 수립해서 수행하죠.
둘째, 다양한 도구를 활용할 수 있습니다. 데이터베이스나 API를 통한 데이터 수집, 파이썬을 활용한 정교한 분석, 시각화 및 문서 작성 등 다양한 도구를 연결해서 자율적으로 작업을 처리합니다.
셋째, 휴먼 인 더 루프 개념을 적용합니다. 중요한 승인 단계나 예상치 못한 예외 상황이 발생하면 인간의 검토와 개입을 요청해서 결과물의 품질과 신뢰성을 보증하는 거죠.
마지막으로 거버넌스 체계를 갖춥니다. 모든 실행 과정의 로그를 기록하고, 템플릿 버전을 관리하며, 철저한 권한 제어를 통해 기업 컴플라이언스를 준수하고 투명성을 확보합니다. 이 네 가지가 AI 에이전트의 핵심 특성이라고 보시면 됩니다.

그렇다면 기존에 많이 사용하던 RPA와는 어떤 차이가 있을까요? 한번 비교해보겠습니다.
RPA는 규칙 기반 자동화(Rule-Based Automation)입니다. 사전에 정의된 규칙과 화면 좌표에 의존하기 때문에 정형화된 GUI 반복 작업에는 강하죠. 하지만 UI가 변경되면 봇이 중단되고, 예외 처리에도 한계가 있습니다. 단일 시스템이나 단순 반복 업무에는 빠르게 도입할 수 있지만, 유지보수 비용이 증가하고 비정형 데이터 처리가 불가능하다는 단점이 있습니다.
반면 AI 에이전트는 목표 지향적이고 적응력이 뛰어납니다. 문서의 의미를 이해하고 레이아웃 변경에도 유연하게 대응하죠. 목표 달성을 위해 필요한 도구를 스스로 호출하고 자기 검토까지 수행합니다. 다양한 API와 데이터 소스 연결이 용이해서 확장성이 뛰어나고, 회복 탄력성이 높으며 복합적인 의사결정을 지원할 수 있습니다.
결론적으로 RPA는 "어떻게"에 집중한다면, AI 에이전트는 "무엇을"에 집중한다고 보시면 됩니다.

이제 에이전트 시스템의 내부 구조를 살펴보겠습니다. 자율적이고 안정적인 보고서 생성을 위해서는 6가지 핵심 컴포넌트가 필요합니다.
첫 번째는 플래너입니다. 목표를 해석하고 계획을 수립하는 역할을 하죠. 큰 목표를 작은 단계로 분해하고, 실행 단계를 생성하며, 전략을 최적화합니다.
두 번째는 오케스트레이터로, 중앙조정과 실행 제어를 담당합니다. 스케줄러를 통해 주기적으로 실행하고, 큐로 작업대기열을 관리하며, 오류 재시도와 장애 복구 롤백도 수행하죠.
세 번째는 리뷰어입니다. 품질 검증과 평가를 수행하는데, 규칙이나 스타일을 검증하고 KPI 임계치를 확인하며, 필요할 때 휴먼 인 더 루프를 요청합니다.
네 번째는 메모리로, 컨텍스트와 지식을 저장합니다. 최근 실행 기록, 보고서 템플릿, 도메인 용어집, 사용자 피드백 등을 보관하죠.
다섯 번째는 실제 실행 도구들입니다. 데이터베이스나 API 연동, 데이터 분석, 시각화, 문서 생성 같은 작업을 수행합니다.
마지막으로 감사 및 거버넌스 컴포넌트가 있습니다. 실행 로그를 남기고, 버전을 관리하며, RBAC 권한 체계를 통해 보안을 확보하죠. 이 6가지가 유기적으로 연결되어 하나의 에이전트 시스템을 구성합니다.

에이전트 시스템을 안전하게 운영하기 위한 6가지 필수 점검 항목을 살펴보겠습니다.
첫째, 데이터 분류입니다. PII, 즉 개인식별정보나 민감정보는 자동 차단 필터를 적용해야 하고, 출력 시에는 마스킹 처리를 해야 합니다.
둘째, 최소 권한 원칙을 따라야 합니다. 데이터베이스나 API 접근 시 읽기와 쓰기 권한을 엄격하게 분리하고, 전용 서비스 계정을 사용하며 키를 철저히 관리해야 하죠.
셋째, 템플릿 무결성을 확보해야 합니다. 템플릿 파일의 해시 검증과 디지털 서명을 적용하고, 승인된 배포 파이프라인을 통해서만 업데이트가 가능하도록 해야 합니다.
넷째, 로그 스키마를 표준화해야 합니다. 입력 프롬프트, 출력 결과, 툴 호출 내역을 전체 로깅하고, 실행 결과 코드와 에러 스택의 추적성을 확보해야 하죠.
다섯째, 비상 차단 기능, 즉 킬 스위치가 필요합니다. 이상 징후 발생 시 에이전트를 즉시 정지시킬 수 있어야 하고, 기능별 토글로 문제 모듈만 격리할 수 있어야 합니다.
마지막으로 규정 준수입니다. ISO 이칠공공일 같은 보안 표준을 준수하는지 정기적으로 감사하고, 생성된 문서와 전자결재 시스템 기록을 연계해야 합니다. 모든 자동화 프로세스는 사내 보안 정책을 최우선으로 준수해야 한다는 점을 기억하세요.

이제 보고서가 어떻게 생성되는지 전체 파이프라인을 살펴보겠습니다. 데이터 소스에서 최종 보고서 배포까지 여섯 단계로 구성됩니다.
1단계는 소스입니다. MES, QMS, EAM 같은 시스템이나 PLC 로그, 엑셀 파일 등에서 데이터가 발생하죠.
2단계는 수집으로, API나 ODBC, SFTP 같은 방법으로 데이터를 가져오고 에어플로우 같은 스케줄러로 주기적으로 실행합니다.
3단계는 정제입니다. 결측값이나 이상치를 처리하고, 코드표를 매핑하며, 데이터를 표준화하죠. 4단계는 분석으로, KPI를 계산하고 트렌드를 파악하며, 이상 감지를 수행합니다.
5단계는 생성입니다. 차트나 표를 시각화하고, 내러티브 텍스트를 생성합니다. 마지막 6단계는 배포단계로, PPT나 PDF, 워드 문서를 생성해서 이메일이나 슬랙을 통해 자동으로 전송합니다. 
이 여섯 단계가 원천 데이터부터 최종 보고서까지의 자동화된 여정입니다.

품질 보고서 자동화의 구체적인 프로세스를 살펴보겠습니다.
먼저 데이터를 집계합니다. 불량 데이터를 일, 주, 월 단위로 자동 수집하고, 전처리 및 이상치를 제거하며, 개선 액션과 담당자를 템플릿에 채워 넣습니다.
다음으로 분석과 시각화를 진행하죠. CTQ나 불량 파레토 차트를 자동으로 생성하고, 상위 5개 결함 항목을 자동으로 추출해서 하이라이트 처리하죠. 원인 가설 텍스트를 엘엘엠을 활용해서 생성합니다.
이제 분석된 결과로 초안을 생성합니다. 모든 분석 결과와 시각화 자료를 하나의 문서로 통합해서 초안을 만들어내는 거죠.
마지막 단계는 승인 및 배포입니다. 휴먼 인 더 루프 방식으로 관리자에게 승인을 요청하고, 승인이 완료되면 이메일이나 메신저를 통해 자동으로 배포됩니다. 이 4단계 흐름이 품질 보고서 자동화의 핵심이죠.

생산 일일 보고는 어떻게 자동화될까요? 역시 4단계로 구성됩니다.
첫 번째는 실적 데이터 취합입니다. MES 시스템과 연동해서 계획 대비 실적 수량을 집계하고, 설비 가동 시간과 정지 시간을 수집하죠.
두 번째는 KPI 자동 계산입니다. 종합설비효율을 산출하고, 가동률과 달성률을 목표 대비로 계산합니다.
세 번째는 이슈 요약 및 알림입니다. 주요 정지나 지연 사유를 로그 텍스트 분석을 통해 요약하고, 기준 미달 시 관리자에게 즉시 알림을 보내죠.
마지막은 보고서 생성입니다. 라인별 하이라이트 슬라이드를 자동으로 생성하고, PDF로 변환해서 공유 폴더에 저장합니다. 이렇게 계획 대비 실적 취합부터 라인별 하이라이트 생성까지 전 과정이 자동화되는 겁니다.

정비 영역의 자동화도 살펴보겠습니다. 설비 이상 수집부터 MTBF, MTTR 갱신까지 4단계로 진행됩니다.
1단계는 데이터 수집입니다. 설비 이상 이벤트를 실시간 로그로 수집하고, 진동이나 온도 트렌드 같은 IoT 센서 데이터를 통합하죠.
2단계는 분석 및 예측입니다. RUL, 즉 잔존 수명을 계산하는 예측 모델을 실행하고, 이상도 스코어를 분석 결과에 반영합니다.
3단계는 작업 연계입니다. 작업지시서를 자동으로 생성하고 담당자를 할당하며, 정비 완료 결과를 시스템 데이터와 연계하죠.
마지막 4단계는 지표 갱신입니다. MTBF와 MTTR을 자동으로 계산해서 갱신하고, 정비 성과 요약 보고서를 자동으로 배포합니다. 이렇게 모니터링부터 AI 분석, 정비 작업, 리포팅까지 전체가 하나의 흐름으로 연결됩니다.

다음은 제조 현장의 핵심 지표들을 실시간으로 모니터링할 수 있는 대시보드 화면 예시입니다.
품질 지표를 먼저 보시면, 총 불량률이 1.2%로 전월 대비 0.3% 개선되었습니다. 초도수율인 FPY는 98.5%이고, PPM은 520이며, 품질비용인 COQ는 12000달러 수준이죠.
생산 지표는 어떨까요? 설비종합효율인 OEE가 85.4%로 2.1% 상승했습니다. 가동률은 92%, 택트 준수율은 99.1%, 납기 준수율은 97.5%로 양호한 편입니다.
정비 지표도 살펴보겠습니다. 평균 고장 간격인 MTBF가 320시간으로 전월 대비 15시간이나 증가되었습니다. 수리 시간인 MTTR은 45분, 설비 가용성은 95.2%, 고장 재발률은 2.0% 수준이죠.
오른쪽 목표 달성 현황을 보시면, 종합 목표 달성률이 92%입니다. 
불량률 감축은 연간 목표인 -20%를 향해 순조롭게 진행 중이고, OEE 향상 목표, +5%는 약간의 리스크가 있으며, MTTR 단축 -10% 목표는 이미 완료되었습니다. 이런 식으로 모든 지표가 전월 대비 성과를 기준으로 실시간 업데이트됩니다.

다음으로 실제 현장에서 적용한 사례를 소개해드리겠습니다. 불량 분석 주간보고 자동화 케이스입니다.
입력 데이터는 세 가지입니다. QMS 불량 내역 테이블을 SQL로 가져오고, 공정이나 라인 마스터 코드표, 그리고 전주 조치 결과를 엑셀로 받아옵니다. 처리 로직은 데이터를 정제하고, 파레토 분석을 수행하며, LLM을 활용해 요약문을 생성하죠.
출력물은 어떤가요? 파레토 차트를 자동으로 그려주고, 찍힘, 스크래치, 이물, 치수 같은, 주간 결함 상위 5개 리스트를 추출합니다. AI가 원인 분석 초안도 생성해주죠.
효과는 놀랍습니다. 기존에 2시간 30분 걸리던 작성 시간이 단 10분으로 줄었어요. 휴먼 에러는 0건으로 완전히 제거되었습니다.
거버넌스 측면에서는 템플릿 버전을 고정해서 임의 수정이 불가능하게 하고, 관리자 승인 후에만 자동 배포되도록 설정했습니다. 반복적인 데이터 취합 업무를 제거하니까 엔지니어들이 정말 중요한 원인 분석과 개선 대책 수립에 집중할 수 있게 되었습니다.

두 번째 사례는 생산 교대조 일일 리포트 자동화입니다. MES 실적 데이터를 기반으로 현장 가시성을 개선한 케이스죠.
입력 데이터는 MES 생산 실적 API, 설비 정지 로그를 PLC ODBC로, 그리고 근무조와 작업자 정보를 HR 데이터베이스에서 가져옵니다. 핵심 로직은 실시간 집계, OEE 계산, 교대 마감 시점의 데이터 스냅샷입니다.
출력물을 보시면 대시보드 카드가 자동으로 생성됩니다. 라인 A의 목표 달성률이 표시되고, 라인 B는 가동정지 상태로 35분간 모터 과열이 발생했다고 나오죠. 라인 C는 생산량이 목표 대비 12% 초과 달성했다고 표시됩니다.
효과는 어떤가요? 보고 속도가 70% 빨라졌고, 가시성은 실시간으로 확보되었습니다. 데이터 수집에 소요되던 시간을 줄이니까 현장 리더가 문제 해결에 즉각 대응할 수 있는 환경이 조성된 거죠.
실시간 알림 기능도 있습니다. OEE가 80% 미만으로 떨어지면 관리자에게 즉시 알림을 발송하고, 불량률 임계치를 초과한 라인은 자동으로 붉은색 표시가 됩니다.

세 번째 사례는 정비 이력 요약 메일 자동화입니다.
입력 데이터는 EAM 작업지시서, 부품 교체 및 재고 사용 기록, 그리고 IoT 설비 센서 알람 로그를 사용합니다. 처리 로직은 MTBF와 MTTR을 계산하고, 이상 징후를 필터링하며, 아웃룩이나 슬랙과 연동하죠.
출력물은 주간 요약 리포트인데, MTBF와 MTTR 추이 그래프, 주요 설비 이슈 상위 3개 요약, 예방 정비 완료율 업데이트가 포함됩니다.
효과는 상당합니다. 주간 정비 회의 준비 시간이 1시간에서 5분으로 단축되었고, 이메일과 슬랙으로 통합 채널이 구축되었죠. 통합 운영 측면에서는 기준치 초과 시 즉시 알림이 발송되고, ERP나 MES 데이터와의 정합성도 검증됩니다.
데이터 수집과 가공 업무가 자동화되니까 엔지니어들이 설비 예방 정비와 개선 활동에 집중할 수 있게 된 겁니다.

이제 실습 파트로 넘어가보겠습니다. CSV 데이터로부터 PPT를 자동으로 생성하는 파이썬 코드를 살펴볼게요.
실습 환경은 파이썬 3.10 이상이 필요하고, pandas, 맷플랏리브, python-pptx 라이브러리를 사용합니다.
입력 데이터를 보시면 날짜, 라인, 결함 유형, 개수가 담긴 CSV 파일입니다. 2024년 12월 1일에 라인 A에서 스크래치가 5건 발생했고, 라인 B에서 찍힘이 두 건 발생했다는 식으로 기록되어 있죠.

전체 코드를 한번 자세히 살펴보겠습니다.
상단에 필요한 라이브러리를 import합니다. 첫 번째 섹션은 데이터 로드 및 집계입니다. read_csv로 데이터를 읽고 날짜를 파싱하며, ISO-calendar.week로 주차를 추출합니다. groupby로 주차와 결함별 count를 합산하고, 전체 기간 합계 기준으로 상위 5개 결함을 선정하죠.
두 번째 섹션은 차트 생성입니다. figure를 6 by 4 사이즈로 만들고, 상위 5개 결함에 대해서만 plot_df를 필터링합니다.
for 루프로 각 결함 유형별로 선 그래프를 그리고, 마커를 설정하며, 레전드를 추가합니다. 
title, x 라벨, y 라벨을 설정하고, tight_layout으로 여백을 조정한 다음 savefig로 150 dpi 해상도로 저장합니다.
세 번째 섹션은 PPT 생성입니다. Presentation 객체를 생성하고, slide_layouts[0]으로 표지 레이아웃을 선택합니다.
title에 '주간 품질 리포트', sub-title에 '자동 생성된 보고서입니다'를 입력하죠. 그다음 slide_layouts[6]으로 빈 레이아웃을 선택하고, add_picture로 차트 이미지를 삽입합니다. 위치는 Inches 함수를 이용해 설정합니다.
마지막에 prs.save로 pptx 파일을 저장하고, 'Saved' 메시지를 출력합니다. 이렇게 50줄 정도의 코드로 자동화가 완성됩니다.

자, 이제 두 번째 섹션으로 넘어가겠습니다. 템플릿 기반 반복업무 절감 사례입니다.
이 섹션의 목표는 템플릿을 설계하고 운영함으로써 보고서 작성 시간을 획기적으로 줄이고 품질을 표준화하는 것입니다. 실제 현장에서 어떻게 적용되는지 구체적인 사례를 통해 알아보죠.

템플릿 기반 자동화가 가져오는 네 가지 핵심 효과를 살펴보겠습니다.
첫째는 표준화입니다. 조직 공통의 레이아웃, 용어, KPI 정의에서 일관성을 확보하게 됩니다. 보고서 간 형식이 통일되니까 부서 간 커뮤니케이션 비용이 자연스럽게 감소하죠.
둘째는 생산성입니다. 반복적인 데이터 취합과 작성 시간을 60%에서 90%까지 절감할 수 있어요. 실무자들이 단순 작업에서 벗어나 데이터 분석과 의사결정 지원에 집중할 수 있게 되는 겁니다.
셋째는 품질 향상입니다. 수기 작성 시 발생하던 누락이나 오탈자를 완전히 제거할 수 있죠. 데이터 정합성이 보장되니까 보고서 신뢰도가 향상되고, 승인 프로세스도 훨씬 빨라집니다.
마지막은 유지보수입니다. 템플릿 파일 기반으로 운영되기 때문에 버전 관리와 변경 추적이 용이합니다. 보고 양식이 변경될 때 코드를 수정할 필요 없이 템플릿만 교체하면 되니까 유연하게 대응할 수 있죠. 이 네 가지가 템플릿 자동화의 핵심입니다.

성공적인 자동화 시스템 구축을 위한 6가지 핵심 가이드라인을 소개하겠습니다.
첫 번째는 데이터 바인딩 패턴입니다. 데이터와 템플릿의 연결점을 명확히 정의해야 합니다. 플레이스홀더 규칙을 통일해서 유지보수성을 높이는 거죠. 
두 번째는 모듈화입니다. 표, 차트, 내러티브 블록 같은 것들을 재사용 가능한 모듈로 설계하면 중복을 줄이고 일관성을 확보할 수 있어요. 헤더 블록, 차트 모듈 같은 식으로 컴포넌트화하는 겁니다.
세 번째는 가드레일, 즉 안전장치입니다. 생성된 내용이 형식을 벗어나지 않도록 제약을 둬야 합니다. 최대 토큰 제한, 포맷 검증 같은 것들로 길이 제한이나 값 형식, 필수 단위를 강제하죠.
네 번째는 지역화 전략입니다. 날짜나 숫자 포맷, 언어 리소스를 로직에서 분리해서 다국어와 지역별 표준을 지원해야 합니다. 
다섯 번째는 접근성입니다. 모든 사용자가 쉽게 읽을 수 있도록 명도 대비, 폰트 크기, 색각 이상 보정 같은 것들을 고려해야 합니다. 고대비 모드 같은 옵션도 제공하는 게 좋습니다.
마지막은 버전 관리와 배포입니다. 템플릿을 코드로 관리하고, 승인된 빌드만 배포해서 템플릿의 무결성을 보장해야 합니다.

재사용성과 유지보수를 위한 블록 설계를 살펴보겠습니다.
먼저, 헤더와 푸터입니다. 문서의 기본 골격이죠. 로고, 문서 제목, 페이지 번호, 날짜, 보안 등급 표시 같은 것들이 포함됩니다.
다음은 왼쪽 블록, KPI 카드입니다. 핵심 지표를 요약해서 보여주는 거죠. 지표명, 현재값, 전월 대비나 전년 대비 증감, 목표 달성률, 상태 아이콘 같은 것들이 들어갑니다.
오른쪽 블록은 차트 모듈입니다. 데이터 시각화를 담당하죠. 표준 색상 팔레트를 적용하고, 축과 범례를 자동으로 조정하며, 주석을 표시합니다.
중심에는 내러티브 블록입니다. 분석과 인사이트, 즉 스토리텔링을 담는 부분이에요. 현황 요약이나 결론, 이슈의 근본 원인 분석, 개선 대책과 담당자 같은 내용이 들어갑니다.
왼쪽 아래는 테이블 모듈입니다. 상세 데이터를 표로 보여주는 건데, 조건부 서식을 적용하고 합계나 평균 행을 추가하며 정렬 규칙을 정의합니다.
마지막으로 오른쪽 아래, 첨부 및 부록입니다. 원본 로그, 현장 사진, 점검표, 참조 링크 같은 것들을 담죠.
문서 구조는 헤더로 시작해서, 요약과 KPI, 컨텍스트인 내러티브와 차트, 그리고 상세 데이터인 테이블과 첨부로 이어지는 흐름입니다.

템플릿 엔진의 핵심인 데이터 바인딩 패턴 6가지를 알아보겠습니다.
첫 번째는 단일 값 바인딩입니다. 가장 단순한 텍스트 치환이죠. 중괄호 두 개로 변수를 감싸면 해당 값이 들어갑니다.
두 번째는 표 반복입니다. 동적으로 행을 추가하고 반복하는 건데, 중괄호 퍼센트로 감싼 for 루프 문법을 사용합니다.
세 번째는 조건부 노출입니다. 상황에 따라 콘텐츠를 보여주거나 숨기는 거죠. for 문과 같이 중괄호 퍼센트로 감싼 if 문으로 제어합니다.
네 번째는 포맷터입니다. 날짜나 숫자 서식을 변환하는 건데, 파이프 기호를 사용합니다.
다섯 번째는 동적 이미지 삽입입니다. 차트나 서명을 자동으로 삽입하는 건데, 인라인 이미지 같은 함수를 사용하죠.
마지막은 다국어입니다. 키 기반으로 언어를 자동 치환하는데, 언더스코어 함수를 사용합니다. 이 6가지 패턴이 Jinja-two 템플릿 엔진의 핵심 기능들입니다.

템플릿 자동화 관점에서 RPA와 AI 에이전트를 비교해보겠습니다.
RPA는 고정 템플릿과 좌표에 의존합니다. 정해진 셀 위치와 규칙에 따라 데이터를 매핑하죠. 문제는 구조 변경에 취약하다는 겁니다. 서식이나 레이아웃이 조금만 바뀌어도 봇 실행이 실패해요. 단순 반복 매핑만 가능하기 때문에 데이터의 의미보다는 위치 일치 여부가 더 중요합니다. UI나 양식이 변경되면 예외 처리 비용이 매우 높아지는 게 한계죠.
반면 AI 에이전트는 의미 기반 바인딩과 컨텍스트를 이해합니다. 문서의 맥락을 파악해서 적절한 위치에 값을 삽입하는 거죠. 유연한 템플릿 적응이 가능해서 서식이 변경되어도 내용을 파악해서 자동으로 조정합니다. 데이터를 단순히 입력하는 걸 넘어서 내용을 요약하고 검토까지 할 수 있죠. 설명 텍스트를 생성하고 변화에 강한 적응력을 갖춘 게 큰 강점입니다.

현장에서 가장 빈번하게 사용되는 표준화된 자동화 템플릿 3종을 소개하겠습니다.
첫 번째는 주간 불량 파레토 보고서입니다. 표준 KPI 카드로 불량률과 PPM을 보여주고, 전체의 80%를 점유하는 상위 결함을 분석하며, 전주 대비 증감을 자동으로 계산합니다. 템플릿 미리보기를 보시면 총 불량 건수가 1,240건으로 증가 추세라고 표시되죠.
두 번째는 수시로 작성하는 고객 클레임 요약입니다. 이슈 발생 타임라인을 시각화하고, 근본원인을 요약하며, 긴급 대책과 영구 대책의 상태를 추적합니다. 미리보기를 보면 10월 1일에 VOC 접수, 10월 2일에 원인 분석 완료, 대책 수립 중이라는 식으로 표시됩니다.
세 번째는 월간 공정능력 보고서입니다. Cp와 Cpk 지수를 자동으로 계산하고, 관리도 트렌드를 보여주며, 공정 개선 제안을 내러티브 형식으로 제공합니다. 미리보기에서 Cp 1.67, Cpk 1.33 같은 지수를 확인할 수 있죠.
템플릿을 모듈화하면 보고서 종류가 늘어나도 유지보수 비용은 최소화됩니다.

생산 주간보고를 워드 문서로 자동 생성하는 사례를 살펴보겠습니다.
입력 데이터는 MES 시스템에서 추출된 생산실적 엑셀 파일과 설비 가동 및 정지 기록이 담긴 다운타임 로그 CSV입니다. 이 두 가지를 원천 데이터로 활용해서 분석을 시작하죠.
템플릿 구성은 어떨까요? 표준 머리말, KPI 지표표, 이슈 요약 블록 같은 것들이 정의된 DOCX 템플릿을 사용합니다. Jinja-two 태그를 활용해서 데이터가 바인딩될 위치를 미리 지정해두는 거죠.
산출물은 완성된 워드 문서입니다. 모든 데이터가 채워진 문서를 PDF로 변환해서 배포하죠.
도입 성과는 어떨까요? 기존에 90분 걸리던 작업이 8분으로 단축되어 91%의 시간을 절감했습니다. 양식 오류는 0건으로 완벽하게 제거되었고요. 자동 배포 덕분에 메일 발송 시간도 대폭 단축되었습니다.
템플릿 기반으로 운영하니까 표준화도 이루어지고, 실무자들은 단순 입력 업무에서 해방되어 분석과 개선에 집중할 수 있게 되었습니다.

정비 보고서 자동화 사례도 살펴보겠습니다.
입력 소스는 EAM 작업지시서와 연동되고, 현장 사진을 자동으로 첨부합니다. 정비 담당자가 작업을 완료하면 시스템에서 자동으로 데이터를 가져오는 거죠.
템플릿 구조는 작업 요약 헤더, 작업 전후 사진 비교, 교체 부품 목록 표, 다음 점검 일정 같은 블록들로 구성됩니다. 각 블록이 모듈화되어 있어서 재사용이 쉽습니다.
자동화 출력물은 정비완료 보고서입니다. 작업 내역, 사진, 부품 정보가 모두 포함된 완전한 문서가 자동으로 생성되죠.

템플릿 자동화를 운영하면서 발생할 수 있는 리스크 시나리오와 대응책을 살펴보겠습니다.
먼저 가장 위험한 리스크는 템플릿 드리프트입니다. 시간이 지나면서 템플릿이 실제 요구사항과 달라지는 현상이죠. 대응책은 분기별로 템플릿을 리뷰하고, 사용자 피드백을 정기적으로 수집하며, 버전 관리 시스템으로 변경 이력을 추적하는 겁니다.
다음으로 위험도가 높은 리스크는 감사 로그 부재입니다. 누가 언제 무엇을 변경했는지 알 수 없으면 문제가 생깁니다. 대응책은 모든 변경사항을 로깅하고, 변경 전후를 비교할 수 있도록 하며, 정기적인 감사를 실시하는 겁니다.
민감정보 유출 리스크의 경우 크리티컬한 리스크입니다. 템플릿에 실수로 민감한 데이터가 포함될 수 있어요. 대응책은 데이터 마스킹 필터를 적용하고, 출력 전에 자동 스캔을 수행하며, RBAC 권한 체계로 접근을 통제하는 겁니다.
중간 수준의 리스크 중 동시성 이슈가 있습니다. 여러 프로세스가 동시에 같은 템플릿을 수정하려고 하면 충돌이 발생하죠. 대응책은 락 메커니즘을 구현하고, 큐 시스템으로 순차 처리를 하며, 재시도 로직을 추가하는 겁니다.
이외에도 잘못된 매핑이나 모델 드리프트와 같은 리스크가 있죠.

이번에는 워드 문서를 자동으로 생성하는 파이썬 실습 코드입니다. 
코드 흐름을 정리하면, 템플릿 로드, 데이터 바인딩, 문서 렌더링, 파일 저장의 4단계입니다. 소요 시간은 약 10분 정도로 빠르게 실습할 수 있습니다.

워드 문서 자동 생성 코드에 대한 자세한 내용은 해당 슬라이드를 참고 바랍니다.

다음은 이메일 자동 발송까지 포함된 RPA 코드입니다. 해당 코드를 참고해서 실습해보시기 바랍니다.

마지막으로 실제 현장에서 에이전트 시스템을 운영할 때 필요한 팁들을 정리했습니다.
첫 번째는 스케줄링입니다. Airflow 같은 워크플로우 엔진을 사용해서, DAG를 정의하고 의존성을 관리하며, 실패 시 재시도 정책을 설정하죠.
두 번째는 로그 추적성입니다. UUID로 각 실행마다 고유 ID를 부여하고, 시작과 종료 타임스탬프를 기록하며, 에러 스택을 저장합니다.
세 번째는 보안/권한입니다. Role-Based Access Control, 즉 역할 기반 접근 제어를 구현해야 합니다. 사용자 그룹별로 권한을 부여하고, 읽기와 쓰기를 분리하며, 민감 정보에 대한 접근을 제한하죠.
네 번째는 템플릿 배포입니다. 템플릿 버전을 태깅하고, 변경사항을 문서화하며, 롤백 계획을 수립합니다. 새 버전을 배포할 때 문제가 생기면 즉시 이전 버전으로 되돌릴 수 있어야 하죠.
