슬라이드 4: 프롬프트 엔지니어링이란? (1분 30초)
그렇다면 프롬프트 엔지니어링이란 정확히 무엇일까요? 단순히 AI에게 질문하는 것을 넘어서는 개념입니다. 이것은 AI 모델의 행동을 설계하고 최적화하는 핵심 기술이에요.

구체적으로는 모델의 역할, 어조, 추론 방식을 정의하여 원하는 결과를 유도하는 기술이죠. 멀티모달 기능과 Function Calling이 확산되면서 복잡한 업무 프로세스 자동화의 핵심 역량으로 부상했습니다. 문서 작성, 코드 생성, 데이터 분석, 고객 지원 등 전 산업 분야에서 필수적인 기술이 되었어요.

슬라이드 5: 역할 정의의 중요성 (2분)
역할 정의는 프롬프트 엔지니어링의 첫 번째 단계입니다. 모델에게 명확한 페르소나를 부여하는 것이 왜 중요할까요?

전문가 페르소나를 설정하면 모델의 정체성이 확립됩니다. "당신은 10년 차 행정 전문가입니다"처럼 구체적으로 명시하는 거죠. 행동과 어조 가이드를 통해 공식적인지 친근한지, 어떤 톤앤매너를 사용할지 결정할 수 있어요.

전문성과 지식 범위를 설정하면 모델이 참고할 도메인 규칙이 명확해집니다. 이렇게 하면 답변의 일관성과 품질이 향상되고, 불필요한 시행착오를 줄일 수 있죠. 결과적으로 도메인 적합도와 신뢰성이 크게 높아집니다.

슬라이드 6: 역할 설계 패턴 (2분)
역할을 설계할 때는 5가지 핵심 요소를 고려해야 합니다. 첫 번째는 System Persona, 즉 정체성입니다. "당신은 10년 차 시니어 파이썬 개발자입니다"처럼 직무와 숙련도를 명시하는 거죠.

두 번째는 Voice & Tone, 어조와 언어 설정입니다. 공식적인 문서인지 친근한 대화인지에 따라 달라져야 해요. 세 번째는 도메인 규칙으로, 특정 분야의 용어집이나 스타일 가이드를 사전에 주입하는 겁니다.

네 번째 도구 사용 기능은 함수 호출이나 데이터베이스 조회 등 외부 도구를 언제 어떻게 사용할지 정의하고요. 마지막 경계 조건은 모델이 하지 말아야 할 행동이나 응답하지 말아야 할 주제를 명확히 제한하는 겁니다.

슬라이드 7: 조건 설정의 범주 (1분 30초)
모델의 출력을 정교하게 제어하려면 4가지 조건 유형을 이해해야 합니다. 출력 형식은 JSON, 표, 마크다운 등 구조적 형태와 길이를 제약하는 거예요. "결과는 반드시 JSON 포맷으로만 출력하고, summary와 bullets 키를 포함하시오"처럼 명확하게 지시합니다.

품질 기준은 근거 제시 여부나 논리적 일관성을 요구하는 것이고요. 리소스 제약은 토큰 수 제한이나 불필요한 서술을 방지하는 겁니다. 마지막으로 정책 및 안전은 개인정보 보호나 편향 방지 같은 윤리 가이드라인을 설정하는 거죠.

슬라이드 8: 왜 공식화가 필요한가 (1분 30초)
프롬프트를 공식화하면 어떤 이점이 있을까요? 첫째는 재현성입니다. 누구나 같은 입력을 주었을 때 일관된 결과를 얻을 수 있죠. 품질의 변동성이 최소화됩니다.

둘째는 협업성이에요. 명확한 공식은 팀원 간 프롬프트 공유와 리뷰, 버전 관리를 용이하게 만듭니다. 셋째, 자동화가 가능해져요. 표준화된 프롬프트는 테스트 파이프라인과 쉽게 통합되고 CI/CD를 실현할 수 있습니다.

마지막으로 확장성입니다. 도메인별로 최적화된 프롬프트를 템플릿 라이브러리로 구축하면 다양한 업무에 신속하게 재사용할 수 있어요. 경험이 아닌 공학적 설계로 예측 가능한 AI 시스템을 만드는 겁니다.

슬라이드 9: 프롬프트 공식 Framework (2분)
이제 실제로 사용할 수 있는 7단계 프롬프트 공식을 소개합니다. 첫 번째 역할(Role)에서 전문가 페르소나를 정의하고요. 두 번째 목표(Objective)에서 수행할 과제를 명시합니다.

세 번째 맥락(Context)은 배경 정보와 상황을, 네 번째 입력(Input)은 처리할 실제 데이터나 문서를 제공하는 단계죠. 다섯 번째 형식(Format)에서 출력 형태를 지정하고, 여섯 번째 제약(Rules)에서 금지 사항과 한계를 설정합니다.

마지막 일곱 번째 평가(Eval)는 결과 검증 기준을 정하는 거예요. 이 흐름은 순차적이지만 반복적일 수 있습니다. 특히 평가 단계에서 결과가 만족스럽지 않다면 역할이나 제약을 수정하며 최적화하는 겁니다.

슬라이드 10: 품질 평가 기준 Rubric (1분 30초)
프롬프트 출력 결과의 품질을 어떻게 측정할까요? 5대 핵심 지표를 사용합니다. 정확성은 환각 없이 사실과 논리에 부합하는지 평가하고요. 완전성은 요청한 모든 세부 사항을 빠짐없이 다루었는지 확인합니다.

형식 준수는 JSON 구조나 특정 태그 사용 등 지정된 포맷을 엄격히 지켰는지 판단하죠. 근거성은 주장에 대한 신뢰할 수 있는 출처나 인용문 포함 여부를 체크합니다. 마지막 안전성은 개인정보나 편향적 표현, 유해 콘텐츠가 필터링되었는지 확인하는 거예요.

이 루브릭은 Python의 assert 문이나 점수 계산 로직으로 변환하여 정량적 평가 시스템을 구축하는 데 활용됩니다.

슬라이드 11: 나쁜 vs 좋은 프롬프트 (1분 30초)
실제 예시를 통해 차이를 비교해볼까요? 나쁜 프롬프트는 "이 문서 요약해줘"처럼 역할, 대상, 형식이 불분명합니다. 모델이 임의로 판단하게 되어 너무 길거나 짧은 요약, 핵심 누락이 발생하죠. 결과물도 비정형 텍스트로 나와 자동화가 불가능합니다.

반면 좋은 프롬프트는 "당신은 행정 문서 전문가입니다. 아래 공문을 5문장 한국어 공식체로 요약하세요"처럼 명확한 페르소나와 작업 지시를 포함해요. 여기에 "핵심 3개 불릿과 근거 문장을 인용하고, JSON 포맷으로만 출력하세요"처럼 구체적인 조건과 출력 형식을 명시합니다.

결과는 구조화된 JSON 데이터로 나와 시스템 후처리가 가능해지는 거죠.

슬라이드 12: 추론 기법 선택지 (2분)
복잡한 문제를 해결하려면 고급 추론 기법이 필요합니다. 첫째, Chain-of-Thought는 "단계별로 생각해보자"와 같이 중간 사고 과정을 명시하여 추론 정확도를 높이는 방법이에요.

둘째, ReAct는 생각과 행동을 교차하며 검색이나 외부 도구를 호출해 사실에 기반한 답을 도출합니다. 셋째, Self-Consistency나 Tree of Thought는 여러 추론 경로를 탐색하여 가장 일관된 답을 선택하거나 생각의 나무를 확장하는 기법이죠.

넷째, Few-shot Prompting은 양질의 예시를 제공하여 In-context Learning을 유도하고요. 마지막 Self-critique는 모델이 생성한 답변을 스스로 비판하고 수정하는 루프를 통해 완성도를 개선합니다. 단순한 지시를 넘어 모델의 사고 과정을 설계하는 겁니다.

슬라이드 13: Python 실습 준비 (1분)
이제 실습으로 넘어가기 전에 개발 환경을 준비해야 합니다. Python 3.10 이상과 Jupyter Notebook 또는 VS Code가 필요해요. 필수 라이브러리는 pip install openai ollama jinja2로 설치하시면 됩니다.

OpenAI API Key를 발급받아 환경변수로 설정하셔야 하고요. 로컬 모델을 활용하시려면 Ollama를 설치한 후 ollama pull llama3로 모델을 다운로드하시면 됩니다. 실습용 예제 텍스트나 CSV 데이터 파일도 미리 준비해주세요.

슬라이드 14: 실습 1 - 기본 메시지 구조 (2분)
첫 번째 실습은 OpenAI API의 기본 사용법입니다. 화면의 코드를 보시면, 먼저 클라이언트를 초기화하고요. 환경변수에서 API Key를 자동으로 로드합니다.

메시지 리스트는 대화의 맥락을 형성하는 부분이에요. system 역할은 모델의 정체성을 설정하고, user 역할은 실제 질문을 담당하죠. "You are a helpful assistant"처럼 간단하게 시작할 수 있습니다.

API 호출에서는 gpt-4o-mini 모델을 지정하고 temperature로 창의성을 조절합니다. 0.2처럼 낮게 설정하면 더 사실적이고 일관된 답변을 얻을 수 있어요. System Role이 프롬프트 엔지니어링의 시작점이라는 것을 기억하세요.

슬라이드 15: 실습 2 - 역할 정의 적용 (2분)
두 번째 실습에서는 구체적인 페르소나를 적용해봅니다. 단순히 "helpful assistant"가 아니라 "대학 행정 및 교육 공문서 작성에 정통한 한국어 기술 커뮤니케이터"처럼 상세하게 정의하는 거죠.

어조는 공식적이고 간결하며 객관적인 문체로, 독자는 국립창원대학교 PRU 재직자 교육생으로 명시합니다. 이런 다중 라인 문자열을 system_role 변수에 저장한 후 role: system의 content로 전달하면 대화의 전반적인 규칙이 됩니다.

사용자 메시지에서는 "3줄로 핵심만 요약해줘"처럼 구체적인 작업과 제약 조건을 제시하고요. temperature를 0.1로 낮게 설정해 사실적 요약을 유도합니다. 전문가 페르소나를 부여하면 불필요한 서론 없이 전문 용어를 적절히 사용한 고품질 결과를 얻을 수 있어요.

슬라이드 16: 실습 3 - JSON 형식 지정 (2분)
세 번째 실습은 출력을 완벽한 JSON 구조로 제어하는 방법입니다. 프롬프트 내에 "JSON으로 출력하라"는 지시와 함께 원하는 Key: Value 구조를 명시해야 해요. "summary와 keywords 배열로 출력하시오"처럼요.

핵심은 API 호출 시 response_format에 type을 json_object로 설정하는 겁니다. 이렇게 하면 강제로 유효한 JSON을 생성하게 되죠. 응답은 문자열 형식이므로 json.loads()를 통해 Python 딕셔너리로 변환하여 사용합니다.

주의할 점은 response_format을 사용할 때 반드시 프롬프트 텍스트 내에도 "JSON"이라는 단어를 포함해야 한다는 거예요. 그렇지 않으면 API가 오류를 반환할 수 있습니다.

슬라이드 17: 실습 4 - 함수 호출 개념 (2분)
네 번째 실습은 Function Calling입니다. LLM이 외부 함수를 실행해야 할 시점을 스스로 판단하도록 유도하는 기법이죠. 먼저 도구를 JSON 스키마 형태로 정의합니다. 함수 이름, 설명, 파라미터 구조를 명시하는 거예요.

"save_summary"라는 함수가 있고, 요약된 결과를 데이터베이스에 저장한다는 설명을 추가합니다. API 호출 시 tools 파라미터에 이 정의를 전달하면 모델이 이를 인지하게 돼요.

중요한 점은 모델이 직접 함수를 실행하지 않는다는 겁니다. 대신 "어떤 함수를 어떤 인자로 실행해야 하는지" JSON으로 응답해요. 개발자는 이 JSON을 파싱하여 실제 코드를 실행하고, 그 결과를 다시 모델에게 알려줘야 합니다.

슬라이드 18: 실습 5 - Few-shot 예제 활용 (2분)
다섯 번째 실습은 Few-shot Prompting입니다. 모델에게 양질의 예시를 미리 보여주어 원하는 형식과 품질을 유도하는 거죠. 입력과 이상적인 출력 쌍을 리스트로 정의합니다.

예를 들어 "2024년도 상반기 교육 일정 안내" 공문에 대해 type은 "안내", keyword는 "교육", urgent는 false라는 JSON을 예시로 보여주는 거예요. 여러 예시를 제공하면 모델이 이 패턴을 모방합니다.

메시지 구조는 System, Examples, New Input 순서로 구성하고요. 이것이 In-context Learning을 유도하는 겁니다. temperature를 0.1로 낮춰 예시 모방을 강화하면 더욱 효과적이에요. 복잡한 지시사항보다 2~3개의 명확한 예시가 훨씬 강력합니다.

슬라이드 19: 로컬 LLM 대안 - Ollama (1분 30초)
클라우드 API 대신 로컬 환경에서 모델을 실행하는 방법도 있습니다. Ollama 라이브러리를 사용하면 OpenAI API와 유사한 인터페이스를 제공해요. 코드 실행 전 터미널에서 ollama pull llama3로 모델 가중치를 다운로드해야 합니다.

메시지 구성은 OpenAI 포맷과 호환되고요. ollama.chat 함수를 바로 호출하여 사용할 수 있습니다. 가장 큰 장점은 데이터가 외부 서버로 전송되지 않아 보안이 우수하고 API 호출 비용이 전혀 발생하지 않는다는 거예요.

민감한 개인정보나 대외비 문서를 처리할 때는 로컬 모델이 가장 안전한 선택지입니다. 인터넷 연결 없이도 동작 가능하죠.

슬라이드 20: 변수화 적용 (1분)
프롬프트를 체계적으로 관리하려면 템플릿 라이브러리가 필요합니다. Python의 f-string이나 Jinja2 템플릿 엔진을 활용하여 동적으로 프롬프트를 생성하는 거죠.

핵심 구성 요소를 모듈화하세요. 역할, 목표, 맥락, 제약 등을 명확히 분리하는 겁니다. Git을 활용하여 변경 이력을 추적하고 패키지 형태로 팀 내에서 공유하면 일관된 품질과 재사용성을 확보할 수 있어요. templates/edu/summary_v1.j2처럼 직관적인 디렉토리 구조를 갖추는 것도 중요합니다.

슬라이드 21: 평가 자동화 (1분 30초)
품질을 정량적으로 평가하려면 자동화 규칙이 필요합니다. 형식 준수는 JSON 스키마 검증으로, 길이 제한은 문장 수나 글자 수 카운트로 자동 체크할 수 있어요.

근거 인용은 정규표현식으로 URL 패턴을 찾아 개수를 세고요. 사실성 검증처럼 복잡한 판단은 GPT-4를 Judge로 사용하는 LLM-based Eval을 활용합니다.

자동화 팁은 단순 형식 검사는 Python 기본 라이브러리로 빠르고 저렴하게 수행하고, 복잡한 항목만 상위 모델을 사용하여 비용 효율적인 평가 파이프라인을 구축하는 겁니다.

슬라이드 22: A/B 테스트 스크립트 (1분 30초)
여러 모델의 성능과 비용을 비교하려면 A/B 테스트가 필요합니다. gpt-4o-mini와 gpt-4o처럼 경량 모델과 고성능 모델을 리스트로 정의하고요.

동일한 프롬프트와 메시지를 사용하여 각 모델에 순차적으로 요청을 보냅니다. 일관성을 위해 temperature는 0으로 설정하는 게 좋아요. 응답 결과를 사전에 정의한 평가 함수로 채점하고 모델별 점수와 응답 내용을 로깅합니다.

이렇게 하면 품질과 속도, 비용을 종합적으로 비교하여 최적의 모델을 선정할 수 있죠.

슬라이드 23: 안전 및 윤리 고려사항 (1분)
프롬프트 엔지니어링에서 안전과 윤리는 필수입니다. PII 마스킹을 통해 개인정보를 보호하고, 편향성을 최소화하기 위한 공정성 체크를 수행해야 해요.

환각, 즉 Hallucination을 완화하기 위해 사실 검증 루프를 구축하고요. 법적 준수를 위해 저작권이나 규정 위반 사항을 필터링해야 합니다. 책임 있는 AI 활용이 무엇보다 중요하죠.

슬라이드 24: Case Study - 도메인 적용 개요 (30초)
이제 실무 적용 사례를 살펴보겠습니다. 교육, 대학 행정, 데이터 분석, 연구 분야에서 프롬프트 엔지니어링이 어떻게 활용되는지 구체적으로 알아볼게요.

슬라이드 25: Case Study 1 - 학생 상담 챗봇 (1분 30초)
첫 번째 사례는 학생 상담 챗봇입니다. 목표는 24시간 학사 및 장학 규정을 안내하는 시스템을 구축하는 거죠. 역할은 "국립창원대학교 학사 전문 상담원"으로 설정하고, 친절하되 정확한 어조를 유지합니다.

제약 조건으로는 허위 정보 생성 금지, 규정집 외 답변 차단, 민감 정보 수집 금지를 명시해요. 출력 형식은 "답변: ...", "근거: 학사규정 제00조" 형태로 구조화합니다.

Function Calling을 활용하여 학사 DB 조회나 일정 확인 기능을 통합할 수도 있죠. 이렇게 설계하면 학생들이 언제든지 정확한 정보를 얻을 수 있는 시스템을 만들 수 있어요.

슬라이드 26: Case Study 2 - 행정 공문 요약기 (1분 30초)
두 번째 사례는 행정 공문 요약기입니다. 목표는 긴 공문을 5문장으로 요약하고 핵심 안건 3가지를 추출하는 겁니다. 페르소나는 "공공기관 공문서 분석 전문가"로 설정하고, 객관적이고 간결한 문체를 사용해요.

JSON 스키마는 summary, action_items 배열, deadline, urgency_level로 구성하고요. 제약 조건으로 원문 문장 직접 인용, 해석이나 추론 금지, 핵심 키워드 볼드 처리를 명시합니다.

실제 공문 텍스트를 input으로 제공하면 구조화된 JSON 결과를 얻을 수 있죠. 이를 행정 업무 자동화 시스템에 통합하면 효율성이 크게 향상됩니다.

슬라이드 27: Case Study 3 - 데이터 분석 보조 (1분 30초)
세 번째 사례는 데이터 분석 보조입니다. 목표는 CSV 데이터를 해석하고 EDA를 제안하며 Python 시각화 코드를 생성하는 거예요. 역할은 "데이터 사이언스 멘토"로 설정하고, 교육적이면서도 실용적인 어조를 유지합니다.

CSV 파일을 입력으로 받아 데이터 구조를 분석하고, 결측치나 이상치 처리 방법을 제안합니다. 시각화 추천은 matplotlib이나 seaborn 코드로 제공하고요.

Few-shot 예시로 "매출 데이터 → 시계열 그래프"처럼 데이터 유형별 분석 방법을 학습시킵니다. 이렇게 하면 비전문가도 데이터 인사이트를 빠르게 얻을 수 있는 시스템을 만들 수 있어요.

슬라이드 28: 협업 워크플로우 (1분)
실무에서는 협업 워크플로우가 중요합니다. 요구 분석부터 프롬프트 설계, 개발, 테스트, 배포, 모니터링, 개선으로 이어지는 생명주기를 관리해야 해요.

Git을 활용한 버전 관리로 프롬프트 변경 이력을 추적하고요. 팀원 간 리뷰와 피드백을 통해 지속적으로 품질을 개선합니다. 문서화와 템플릿 공유로 조직 차원의 역량을 축적하는 것이 핵심이죠.

슬라이드 29: 프롬프트 디버깅 체크리스트 (1분)
프롬프트가 제대로 작동하지 않을 때 체크해야 할 5가지 항목입니다. 첫째, 모호성 제거입니다. 역할이나 목표가 명확한지 확인하세요. 둘째는 조건 누락으로, 형식이나 제약이 빠지지 않았는지 점검하고요.

셋째, 반례 테스트로 엣지 케이스를 시험해봅니다. 넷째, 추론 기법 적용 여부를 확인하고요. 다섯째, 로그 축적을 통해 실패 사례를 분석하고 패턴을 찾아냅니다. 이 체크리스트를 습관화하면 디버깅 시간을 크게 줄일 수 있어요.
