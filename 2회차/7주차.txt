문서의 레이아웃은 템플릿으로 고정하고, 변수나 플레이스홀더에 Excel, CSV, 데이터베이스, API에서 가져온 데이터를 바인딩하면 끝입니다. 이렇게 하면 일관성 있는 고품질 문서를 대량으로 빠르게 생성할 수 있죠.

슬라이드 10: 템플릿 설계 원칙
견고한 자동화 템플릿을 만들기 위한 4가지 핵심 원칙이 있습니다. 첫째, 명명 규칙은 snake_case를 사용하고 일관성을 유지하세요.

둘째, 변수는 최소화하고 재사용성을 최대화합니다. 셋째, 예외 상황이나 선택적 필드는 if문이나 for문으로 처리하는 전략을 수립하세요. 마지막으로 Git 같은 버전 관리 시스템을 사용하고, 항상 샘플 데이터를 함께 보관하는 것이 중요합니다.

슬라이드 11: 템플릿 엔진/도구 비교
Python에서 문서 자동화에 사용하는 주요 라이브러리들을 비교해보겠습니다. 표를 보시면 각 도구의 용도와 장점이 정리되어 있어요.

Jinja2는 문자열과 마크업 처리에 직관적이고, DocxTpl은 Word 템플릿에서 스타일을 보존하면서 Jinja2 문법을 쓸 수 있습니다. python-pptx는 슬라이드 자동화에 최적이고, pandas와 openpyxl은 엑셀 데이터 읽기와 전처리에 필수적이죠. 실습에서 이 도구들을 직접 사용해보실 거예요.

슬라이드 12: 산출물 포맷 전략
산출물 포맷은 업무 목적에 따라 전략적으로 선택해야 합니다. Word는 편집이 필요한 상세 보고서나 계약서에 적합하고, PowerPoint는 발표용 자료에 최적이죠.

outputs 폴더는 생성된 문서가 저장되는 곳이고, src에는 Python 스크립트를 작성합니다. 중요한 건 .env 파일인데요, API 키나 민감한 설정은 여기에 넣고 반드시 gitignore에 등록해서 코드 저장소에 올라가지 않도록 해야 합니다.

슬라이드 19: 템플릿 설계 - Jinja2 문법
Word 문서에 Python 데이터를 심기 위해 Jinja2 템플릿 태그를 사용합니다. 화면에 보이는 것처럼 이중 중괄호로 변수를 표시하고, {% %}로 제어문을 작성하는 거예요.

예를 들어 {{ project_name }}은 프로젝트명 변수를 출력하고, {% for item in risks %}는 리스크 목록을 반복하면서 표를 채웁니다. 조건문도 사용할 수 있어요. {% if summary %}처럼 특정 필드가 있을 때만 내용을 보여주도록 설정할 수 있죠.

슬라이드 20: 코드 - 엑셀 로딩 + 템플릿 채우기
실제 Python 코드를 보겠습니다. 왼쪽 화면의 코드가 핵심 로직이에요.

먼저 pandas로 Excel 파일을 읽어서 데이터를 정제합니다. 그다음 DocxTemplate 객체로 템플릿을 로드하고, render 메서드에 데이터 딕셔너리를 전달해서 변수를 채워 넣죠. 파일명은 프로젝트명을 슬러그화해서 안전하게 만들고, 저장 경로가 없으면 자동으로 생성합니다. 마지막으로 try-except 블록으로 예외 처리를 하고 로그를 기록하면 안정적인 스크립트가 완성돼요.

슬라이드 23: 목표/API 및 보안
이번 실습의 핵심은 AI 요약 생성과 데이터 보안의 균형점을 찾는 것입니다. LLM에게 회의록이나 긴 텍스트를 요약하게 하고, 그 결과를 템플릿에 자동으로 넣을 거예요.

중요한 건 보안입니다. API 키는 환경변수로 보관하고, 개인정보는 반드시 마스킹 처리를 하세요. 프롬프트에 비밀정보나 민감한 데이터를 직접 넣지 않도록 각별히 주의해야 합니다.

슬라이드 24: 프롬프트 템플릿 설계
구조적 프롬프트 전략을 3단계로 나눠보겠습니다. 코드를 보시면 먼저 시스템 프롬프트에서 AI의 역할과 제약을 명확히 정의하고 있어요.

두 번째로 사용자 프롬프트 템플릿에는 데이터를 주입할 플레이스홀더를 넣습니다. 세 번째가 중요한데요, JSON Schema로 출력 구조를 강제하면 자연어 응답이 아닌 파싱 가능한 데이터를 받을 수 있습니다. summary, risk_factors, sentiment 같은 키가 보장되니까 후속 Python 코드에서 에러 없이 바로 사용할 수 있죠.

슬라이드 25: 코드 - API 호출 + 문서화
AI API를 호출하고 결과를 문서화하는 전체 코드입니다. tenacity 라이브러리로 재시도 로직과 백오프 전략을 구현했어요.

API 응답이 오면 JSON으로 파싱하고, pydantic으로 데이터 유효성을 검증합니다. 검증이 통과되면 DocxTpl에 컨텍스트를 바인딩해서 최종 문서를 저장하는 거죠. 이렇게 하면 AI의 창의성과 자동화의 안정성을 모두 확보할 수 있어요.

슬라이드 26: Section 6 - 실습3 시작
세 번째 실습입니다. 이제 대량 문서를 자동으로 생성하는 워크플로우를 설계해볼 건데요.

폴더를 실시간으로 감시하다가 새 파일이 들어오면 자동으로 문서를 생성하고, 에러가 발생하면 재처리 큐에 넣는 실무급 파이프라인을 만들어보겠습니다.

슬라이드 27: 파이프라인 설계
4단계 워크플로우를 보시죠. 첫 번째로 Watchdog 라이브러리로 폴더를 감시해서 신규 Excel이나 JSON 파일을 감지합니다.

두 번째 단계에서 템플릿 엔진으로 문서를 생성하고, 세 번째로 데이터 무결성을 검증합니다. 마지막으로 완성된 파일을 지정된 경로에 배포하고 로그를 기록하는 거예요. 동시성 처리는 ThreadPool을 사용하면 여러 파일을 병렬로 처리할 수 있습니다.

슬라이드 28: 코드 - 감시 및 에러 핸들링
실제 구현 코드를 살펴보겠습니다. watchdog의 FileSystemEventHandler를 상속받아서 파일 생성 이벤트를 처리하는 클래스를 만들었어요.

구조적 로깅을 위해 logging 모듈을 사용하고, JSON formatter를 적용하면 나중에 로그 분석이 쉽습니다. 실패한 작업은 재시도 큐에 넣고, 일정 횟수 이상 실패하면 에러 폴더로 이동시켜서 수동 확인할 수 있도록 하는 게 안정적인 운영 방법입니다.

슬라이드 29: Section 7 - 실무 사례
마지막 섹션입니다. 지금까지 배운 내용을 실제 업무에 어떻게 적용했는지 사례를 통해 알아보겠습니다.

기획서, 회의록, 정기 보고서 자동화 성공 사례와 함께 실무에서 꼭 지켜야 할 베스트 프랙티스를 공유하겠습니다.

슬라이드 30: 사례 1·2 - 기획서/회의록
왼쪽에는 입력 데이터와 프로세스가 보입니다. 프로젝트명, 예산, 일정표 같은 데이터를 Excel로 준비하면 템플릿 엔진이 표준 기획서로 변환해주는 거예요.

오른쪽은 효과입니다. 작성 시간이 70% 감소했고, 필수 항목 누락이 거의 사라졌어요. 게다가 모든 버전이 Git으로 관리되니까 이력 추적도 완벽합니다. 회의록도 마찬가지로 아젠다와 참석자, 결정사항만 입력하면 표준 포맷의 회의록이 자동 생성됩니다.

슬라이드 31: 사례 3·4 - 정기 보고/성과 리포트
정기 보고서는 데이터 소스가 다양합니다. ERP에서 재무 지표를, CRM에서 영업 파이프라인을, Excel에서 KPI 추적 데이터를 가져와서 통합할 수 있어요.

핵심은 지표 표준화입니다. 매번 같은 기준으로 데이터를 집계하고, AI가 요약 코멘트를 생성하며, 차트는 미리 정의된 자리표에 자동으로 들어가죠. 이렇게 하면 월말마다 몇 시간씩 걸리던 보고서 작성이 클릭 한 번으로 끝납니다.

슬라이드 32: Best Practice 체크리스트
성공적인 자동화 운영을 위한 핵심 체크리스트입니다. 배포 전에 반드시 확인하세요.

먼저 템플릿과 데이터 사이의 계약서를 만들어서 필수 필드와 선택 필드를 명확히 정의하세요. 샘플 데이터와 골든 셋은 항상 유지해야 테스트가 쉽습니다. 롤백 가능한 배포 전략과 버전 태깅도 중요하고, 보안 측면에서는 API 키 관리, 로그 마스킹, 민감정보 처리 방침을 확립해야 합니다.

슬라이드 33: 정리 - 오늘의 핵심
